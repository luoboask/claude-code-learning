# 记忆读取与验证

> 🔍 如何正确读取和验证记忆 | 防止过时记忆误导

---

## 🎯 学习目标

完成本节后，你将能够：
- ✅ 知道何时访问记忆
- ✅ 掌握读取时验证机制
- ✅ 处理记忆冲突
- ✅ 避免记忆漂移问题

## 1. 何时访问记忆

### 1.1 应该访问的情况

```markdown
## When to access memories

✅ 应该访问:

1. 当记忆看起来相关时
   - 用户话题与记忆内容相关
   - 需要上下文来理解请求

2. 当用户引用 prior-conversation 工作时
   - "上次我们讨论的..."
   - "记得我之前说过..."

3. 当用户明确要求检查、回忆或记住时
   - "检查记忆"
   - "回忆一下..."
   - "你记得什么？"
```

### 1.2 不应该访问的情况

```markdown
❌ 不应该访问:

如果用户说"忽略"或"不使用"记忆：
- 像 MEMORY.md 为空一样处理
- 不要应用记忆事实
- 不要引用记忆
- 不要对比记忆
- 不要提及记忆内容

示例:
用户："忽略记忆，我们重新开始"
助手：[像没有记忆一样继续对话]
```

### 1.3 访问决策树

```
用户请求
    │
    ├─→ 用户说"忽略记忆"或"不使用记忆"
    │       └─→ 不访问，像 MEMORY.md 为空处理
    │
    ├─→ 用户明确要求检查/回忆
    │       └─→ 访问记忆
    │
    ├─→ 记忆看起来相关
    │       └─→ 访问记忆
    │
    └─→ 无明显相关
            └─→ 可选访问（根据上下文判断）
```

## 2. 读取时验证

### 2.1 为什么需要验证？

**记忆是快照，不是实时数据**。

```
记忆记录的是"当时为真"的信息，可能随时间过时。

"The memory says X exists" is not the same as "X exists now."
```

### 2.2 验证原则

```markdown
## Before recommending from memory

A memory that names a specific function, file, or flag is a claim that it 
existed *when the memory was written*. It may have been renamed, removed, or 
never merged. Before recommending it:

1. **如果记忆命名文件路径**: 检查文件存在
2. **如果记忆命名函数或标志**: grep 搜索
3. **如果用户即将基于推荐行动**: 先验证
```

### 2.3 验证流程

```
读取记忆
    │
    ↓
┌─────────────────────────────────┐
│ 1. 检查记忆类型                  │
│ - 是否命名具体文件/函数/标志？   │
└───────────────┬─────────────────┘
                │
                ↓
          ┌─────┴─────┐
          │           │
       [是]        [否]
          │           │
          ↓           ↓
┌──────────────┐  ┌──────────────┐
│ 2. 验证存在性 │  │ 可直接使用   │
│ - 检查文件    │  │              │
│ - grep 搜索   │  │              │
└──────┬───────┘  └──────────────┘
       │
       ↓
┌─────────────────────────────────┐
│ 3. 验证结果                      │
│ - 存在 → 可推荐                 │
│ - 不存在 → 更新/删除记忆        │
│ - 冲突 → 优先相信当前观察       │
└─────────────────────────────────┘
```

### 2.4 验证命令

```bash
# 1. 检查文件路径
test -f /path/to/file && echo "存在" || echo "不存在"

# 2. 搜索函数
grep -r "function_name" src/

# 3. 搜索标志
grep -r "FEATURE_FLAG" config/

# 4. 检查 Git 状态
git log --oneline -10

# 5. 读取当前代码
cat /path/to/file
```

## 3. 记忆漂移处理

### 3.1 漂移警告

```markdown
⚠️ 记忆漂移警告:

Memory records can become stale over time. Use memory as context for what was 
true at a given point in time. Before answering the user or building assumptions 
based solely on information in memory records, verify that the memory is still 
correct and up-to-date by reading the current state of the files or resources. 
If a recalled memory conflicts with current information, trust what you observe 
now — and update or remove the stale memory rather than acting on it.
```

### 3.2 漂移示例

```markdown
# 过时记忆示例

## 记忆内容 (2026-01-01 保存)
---
name: 认证中间件
description: 项目使用旧认证中间件
type: project
---

项目使用旧认证中间件处理用户认证。

## 当前状态 (2026-04-02)
项目已重写认证中间件（法律合规要求）。

## 处理
1. 发现记忆与当前状态冲突
2. 优先相信当前观察（阅读代码）
3. 更新记忆：
   ---
   name: 认证中间件重写
   description: 认证中间件已重写，满足法律合规要求
   type: project
   updated: 2026-04-02
   ---
   
   认证中间件已重写，满足法律合规要求。
   旧中间件已移除。
```

### 3.3 漂移处理原则

```markdown
✅ 应该做:

1. 优先相信当前观察
   - 阅读代码
   - 检查文件
   - 运行命令

2. 更新或移除过时记忆
   - 发现冲突时立即更新
   - 不再适用的记忆删除

3. 记录变更原因
   - 在记忆中说明为什么更新

❌ 不应该做:

1. 盲目相信过时记忆
2. 忽视当前观察
3. 保留冲突记忆
```

## 4. 记忆搜索

### 4.1 搜索方法

```markdown
## Searching past context

当寻找过去上下文时：

1. 搜索记忆目录中的主题文件:
   ```bash
   # 使用 GrepTool 或 shell grep
   grep -rn "<search term>" ~/.claude/projects/<slug>/memory/ --include="*.md"
   ```

2. 会话转录日志（最后手段 — 大文件，慢）:
   ```bash
   grep -rn "<search term>" <project-dir>/ --include="*.jsonl"
   ```
```

### 4.2 搜索策略

```
搜索请求
    │
    ↓
┌─────────────────────────────────┐
│ 1. 搜索记忆文件                 │
│ - grep -rn "<term>" memory/     │
└───────────────┬─────────────────┘
                │
                ↓
          ┌─────┴─────┐
          │           │
      [找到]      [未找到]
          │           │
          ↓           ↓
┌──────────────┐  ┌──────────────┐
│ 2. 验证内容   │  │ 2. 搜索转录  │
│ - 检查相关性 │  │ - grep *.jsonl│
│ - 检查过时   │  │ - 最后手段   │
└──────┬───────┘  └──────┬───────┘
       │                 │
       ↓                 ↓
┌─────────────────────────────────┐
│ 3. 返回结果                      │
│ - 相关记忆内容                   │
│ - 或转录片段                     │
└─────────────────────────────────┘
```

### 4.3 搜索示例

```bash
# 示例 1: 搜索用户角色
grep -rn "数据科学家" ~/.claude/projects/my-project/memory/ --include="*.md"
# 输出：memory/user_role.md:5:用户是数据科学家

# 示例 2: 搜索反馈
grep -rn "mock 数据库" ~/.claude/projects/my-project/memory/ --include="*.md"
# 输出：memory/feedback_testing.md:3:集成测试禁止 mock 数据库

# 示例 3: 搜索项目信息
grep -rn "冻结" ~/.claude/projects/my-project/memory/ --include="*.md"
# 输出：memory/project_freeze.md:3:2026-03-05 后冻结合并
```

## 5. 记忆冲突处理

### 5.1 冲突类型

| 类型 | 示例 | 处理 |
|------|------|------|
| **记忆 vs 当前代码** | 记忆说函数存在，grep 找不到 | 优先相信 grep，更新记忆 |
| **记忆 vs 当前状态** | 记忆说项目用旧中间件，代码已重写 | 优先相信代码，更新记忆 |
| **记忆 vs Git 历史** | 记忆说 X 负责，git blame 显示 Y | 优先相信 Git，更新记忆 |
| **记忆 vs 记忆** | 两个记忆冲突 | 检查日期，优先更新，或询问用户 |

### 5.2 冲突处理流程

```
发现冲突
    │
    ↓
┌─────────────────────────────────┐
│ 1. 确认冲突                      │
│ - 验证当前状态                   │
│ - 检查记忆日期                   │
└───────────────┬─────────────────┘
                │
                ↓
┌─────────────────────────────────┐
│ 2. 决定信任源                    │
│ - 优先相信当前观察               │
│ - 优先相信权威源（Git 等）       │
└───────────────┬─────────────────┘
                │
                ↓
┌─────────────────────────────────┐
│ 3. 更新记忆                      │
│ - 修正错误记忆                   │
│ - 删除过时记忆                   │
│ - 合并冲突记忆                   │
└───────────────┬─────────────────┘
                │
                ↓
┌─────────────────────────────────┐
│ 4. 记录变更                      │
│ - 添加 updated 字段             │
│ - 说明变更原因                   │
└─────────────────────────────────┘
```

### 5.3 冲突示例

```markdown
# 示例：记忆与代码冲突

## 记忆内容
---
name: 认证函数
description: 使用 authenticateUser() 函数
type: reference
---

用户认证使用 authenticateUser() 函数。

## 当前代码
$ grep -r "authenticateUser" src/
# 无结果

$ grep -r "verifyUser" src/
src/auth.ts: export function verifyUser() {...}

## 处理
1. 发现记忆说 authenticateUser()，但代码中是 verifyUser()
2. 优先相信 grep 结果
3. 更新记忆：
   ---
   name: 认证函数
   description: 使用 verifyUser() 函数
   type: reference
   updated: 2026-04-02
   ---
   
   用户认证使用 verifyUser() 函数。
   
   **变更原因:** 函数已重命名（2026-03-15 重构）
```

## 6. 最佳实践

### 6.1 读取记忆

```markdown
✅ 应该做:

1. 先搜索再读取
   - grep 搜索相关记忆
   - 检查多个相关文件

2. 验证后再推荐
   - 命名文件/函数 → 检查存在性
   - 即将行动 → 先验证

3. 注意记忆日期
   - 检查 created/updated 字段
   - 评估是否可能过时

4. 交叉验证
   - 记忆 vs 代码
   - 记忆 vs Git
   - 记忆 vs 当前状态

❌ 不应该做:

1. 盲目相信记忆
2. 不验证就推荐
3. 忽视当前观察
4. 保留冲突记忆
```

### 6.2 响应用户

```markdown
# ✅ 好的响应

用户："我记得有个 authenticateUser 函数，在哪？"

助手：
"根据记忆，你提到过 authenticateUser 函数。
但我刚刚搜索了代码库，没有找到这个函数。
找到的是 verifyUser() 函数（在 src/auth.ts）。

可能是：
1. 函数已重命名
2. 记忆过时了

你想查看 verifyUser() 函数吗？我可以更新记忆。"

# ❌ 坏的响应

用户："我记得有个 authenticateUser 函数，在哪？"

助手：
"authenticateUser 函数在记忆中提到过。
[不验证就直接推荐，可能误导用户]"
```

## 7. 用户要求忽略记忆

### 7.1 处理流程

```
用户说"忽略记忆"
    │
    ↓
┌─────────────────────────────────┐
│ 1. 确认用户意图                  │
│ - 是真的要忽略还是临时？         │
└───────────────┬─────────────────┘
                │
                ↓
┌─────────────────────────────────┐
│ 2. 像 MEMORY.md 为空处理         │
│ - 不读取记忆                     │
│ - 不引用记忆                     │
│ - 不应用记忆事实                 │
└───────────────┬─────────────────┘
                │
                ↓
┌─────────────────────────────────┐
│ 3. 继续对话                      │
│ - 基于当前上下文                 │
│ - 不提及记忆                     │
└─────────────────────────────────┘
```

### 7.2 示例

```markdown
# 示例 1: 临时忽略

用户："暂时忽略记忆，我们试试另一种方法"

助手：
"好的，我们暂时忽略记忆，试试另一种方法。
[继续对话，不引用记忆]"

# 示例 2: 永久忽略

用户："忘记我之前说的测试要用真实数据库，我们改回 mock"

助手：
"好的，我会更新记忆。
[找到 feedback_testing.md，更新或删除]"
```
