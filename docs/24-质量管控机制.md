## 1. 为什么需要质量管控？

### 1.1 问题背景

```
没有质量管控 → 记忆文件无限膨胀 → 加载慢、检索难、质量低
```

**具体问题**:
- 记忆文件可能增长到数百 KB
- 加载时间变长
- 检索效率降低
- AI 倾向于写冗长内容

### 1.2 解决方案

```
双重限制:
1. 行数限制：MAX_ENTRYPOINT_LINES = 200
2. 字节限制：MAX_ENTRYPOINT_BYTES = 25,000 (~25KB)

目标:
- 强制 AI 写精简内容
- 保持加载和检索效率
- 提高记忆质量
```

## 2. 双重限制机制

### 2.1 限制定义

```typescript
// source/src/memdir/memdir.ts

export const ENTRYPOINT_NAME = 'MEMORY.md'
export const MAX_ENTRYPOINT_LINES = 200
export const MAX_ENTRYPOINT_BYTES = 25_000  // ~25KB
```

### 2.2 限制选择原因

| 限制 | 值 | 原因 |
|------|-----|------|
| **行数** | 200 | 约 100-150 条记忆索引 |
| **字节** | 25KB | 约 125 字符/行 × 200 行 |

### 2.3 双重限制的优势

```
单一行数限制的问题:
- 长行可能超过 1KB
- 200 行 × 1KB = 200KB（太大）

单一字节限制的问题:
- 可能只有几行超长内容
- 不利于索引组织

双重限制的优势:
- 既限制行数，又限制总大小
- 强制每行精简（<150 字符）
- 保持索引结构清晰
```

## 3. 截断逻辑

### 3.1 截断流程

```typescript
// source/src/memdir/memdir.ts

export function truncateEntrypointContent(raw: string): EntrypointTruncation {
  const trimmed = raw.trim()
  const contentLines = trimmed.split('\n')
  const lineCount = contentLines.length
  const byteCount = trimmed.length

  // 检查是否需要截断
  const wasLineTruncated = lineCount > MAX_ENTRYPOINT_LINES
  const wasByteTruncated = byteCount > MAX_ENTRYPOINT_BYTES

  if (!wasLineTruncated && !wasByteTruncated) {
    return { content: trimmed, lineCount, byteCount, wasLineTruncated, wasByteTruncated }
  }

  // 1. 先按行截断（自然边界）
  let truncated = wasLineTruncated
    ? contentLines.slice(0, MAX_ENTRYPOINT_LINES).join('\n')
    : trimmed

  // 2. 再按字节截断（在最后一个换行处）
  if (truncated.length > MAX_ENTRYPOINT_BYTES) {
    const cutAt = truncated.lastIndexOf('\n', MAX_ENTRYPOINT_BYTES)
    truncated = truncated.slice(0, cutAt > 0 ? cutAt : MAX_ENTRYPOINT_BYTES)
  }

  // 3. 生成警告信息
  const reason =
    wasByteTruncated && !wasLineTruncated
      ? `${formatFileSize(byteCount)} (limit: ${formatFileSize(MAX_ENTRYPOINT_BYTES)}) — index entries are too long`
      : wasLineTruncated && !wasByteTruncated
        ? `${lineCount} lines (limit: ${MAX_ENTRYPOINT_LINES})`
        : `${lineCount} lines and ${formatFileSize(byteCount)}`

  return {
    content: truncated +
      `\n\n> WARNING: ${ENTRYPOINT_NAME} is ${reason}. Only part of it was loaded.`,
    lineCount,
    byteCount,
    wasLineTruncated,
    wasByteTruncated,
  }
}
```

### 3.2 截断顺序

```
1. 先按行截断
   - 在自然边界（换行符）处截断
   - 保留完整行

2. 再按字节截断
   - 在最后一个换行处截断
   - 避免切断半行

3. 添加警告
   - 说明哪种限制触发
   - 提醒用户保持精简
```

### 3.3 截断示例

```markdown
# 示例 1: 行数超限

## 原始内容 (250 行)
- [记忆 1](file1.md) — 描述
- [记忆 2](file2.md) — 描述
...
- [记忆 250](file250.md) — 描述

## 截断后 (200 行 + 警告)
- [记忆 1](file1.md) — 描述
- [记忆 2](file2.md) — 描述
...
- [记忆 200](file200.md) — 描述

> WARNING: MEMORY.md is 250 lines (limit: 200). Only part of it was loaded.
> Keep index entries to one line under ~150 characters; move detail into topic files.
```

```markdown
# 示例 2: 字节超限

## 原始内容 (30KB)
- [记忆 1](file1.md) — 这是一行非常非常非常长的描述，超过了 150 字符限制...
- [记忆 2](file2.md) — 同样超长的描述...
...

## 截断后 (25KB + 警告)
- [记忆 1](file1.md) — 这是一行非常非常非常长的描述，超过了 150...
- [记忆 2](file2.md) — 同样超长的描述，超过了 150...
...

> WARNING: MEMORY.md is 30KB (limit: 25KB) — index entries are too long.
> Keep index entries to one line under ~150 characters; move detail into topic files.
```

## 4. 警告信息

### 4.1 警告格式

```markdown
> WARNING: MEMORY.md is {reason}. Only part of it was loaded.
```

### 4.2 警告类型

| 触发条件 | 警告信息 |
|----------|----------|
| **仅行数超限** | `250 lines (limit: 200)` |
| **仅字节超限** | `30KB (limit: 25KB) — index entries are too long` |
| **两者都超限** | `250 lines and 30KB` |

### 4.3 警告示例

```markdown

# 示例 3: 两者都超限
```

## 5. 保持记忆精简

### 5.1 索引条目规范

```markdown
# ✅ 好的索引条目（<150 字符）

- [测试必须用真实数据库](feedback_testing.md) — 集成测试禁止 mock，去年生产事故
  # 102 字符

- [数据科学家角色](user_role.md) — 专注于可观测性/logging
  # 58 字符

- [移动端发布冻结](project_freeze.md) — 2026-03-05 后冻结
  # 65 字符

# ❌ 坏的索引条目（>150 字符）

- [测试必须用真实数据库](feedback_testing.md) — 集成测试禁止 mock 数据库，必须使用真实连接，因为去年 mock 测试通过但生产迁移失败，导致 2 小时停机，这是一个项目测试政策，每个贡献者都应该遵循
  # 180 字符，太长！
```

### 5.2 精简技巧

```markdown
技巧 1: 使用缩写
❌ 集成测试必须使用真实数据库连接，不能使用 mock 数据库
✅ 集成测试禁 mock，必须真实数据库

技巧 2: 省略冗余词
❌ 用户是数据科学家，目前专注于可观测性和日志分析领域
✅ 数据科学家，专注可观测性/logging

技巧 3: 使用符号
❌ 截止日期是 2026 年 3 月 5 日
✅ 截止 2026-03-05

技巧 4: 详情放入主题文件
❌ 在索引中写完整内容
✅ 索引只写钩子，详情在主题文件
```

### 5.3 主题文件结构

```markdown
# 主题文件 (feedback_testing.md)

---
name: 测试必须用真实数据库
description: 集成测试禁止 mock 数据库，必须使用真实连接
type: feedback
scope: team
---

集成测试必须 hit 真实数据库，不能使用 mock。

**Why:** 
去年 Q4 发布时，mock 的数据库测试全部通过，但生产迁移因字段类型不匹配失败，
导致 2 小时停机。根本原因：mock 层没有捕获 schema 变更。

**How to apply:**
- 所有集成测试必须配置真实数据库连接
- 使用测试数据库（非生产）
- CI/CD 配置中启用真实数据库
- 单元测试可以 mock
- 端到端测试必须真实环境

# MEMORY.md 索引

- [测试必须用真实数据库](feedback_testing.md) — 集成测试禁 mock，去年生产事故
```

## 6. 日志记录

### 6.1 异步日志

```typescript
// source/src/memdir/memdir.ts

function logMemoryDirCounts(memoryDir: string, baseMetadata: Record<string, any>): void {
  const fs = getFsImplementation()
  void fs.readdir(memoryDir).then(
    dirents => {
      let fileCount = 0
      let subdirCount = 0
      for (const d of dirents) {
        if (d.isFile()) fileCount++
        else if (d.isDirectory()) subdirCount++
      }
      logEvent('tengu_memdir_loaded', {
        ...baseMetadata,
        total_file_count: fileCount,
        total_subdir_count: subdirCount,
      })
    },
    () => {
      // 目录不可读，记录不含计数的日志
      logEvent('tengu_memdir_loaded', baseMetadata)
    },
  )
}
```

### 6.2 记录事件

```typescript
logEvent('tengu_memdir_loaded', {
  content_length: 25000,      // 内容长度
  line_count: 200,            // 行数
  was_truncated: true,        // 是否被截断
  was_byte_truncated: false,  // 是否被字节截断
  memory_type: 'auto',        // 记忆类型
  total_file_count: 15,       // 文件数
  total_subdir_count: 0,      // 子目录数
})
```

## 7. 最佳实践

### 7.1 写入时

```markdown
✅ 应该做:

1. 保持索引条目 <150 字符
   - 使用缩写
   - 省略冗余词
   - 使用符号

2. 详情放入主题文件
   - 索引只写钩子
   - 主题文件写完整内容

3. 定期检查 MEMORY.md
   - 检查是否接近限制
   - 合并重复条目
   - 删除过时条目

❌ 不应该做:

1. 在索引中写完整内容
2. 使用冗长描述
3. 忽视警告信息
```

### 7.2 超限时

```markdown
✅ 应该做:

1. 检查哪些条目超长
   - 找出>150 字符的条目

2. 精简超长条目
   - 使用缩写
   - 省略冗余词

3. 合并重复条目
   - 相关记忆合并

4. 删除过时条目
   - 不再适用的记忆

❌ 不应该做:

1. 忽视警告
2. 继续添加新条目
3. 不精简现有条目
```

## 8. 常见问题

### Q1: 为什么是 200 行/25KB？

**A**: 这是平衡点：
- 200 行 ≈ 100-150 条记忆
- 25KB ≈ 125 字符/行 × 200 行
- 足够大多数项目使用
- 保持加载效率

### Q2: 超限时怎么办？

**A**: 
1. 精简现有条目
2. 合并重复记忆
3. 删除过时记忆
4. 详情移入主题文件

### Q3: 截断会影响记忆使用吗？

**A**: 
- 截断的只是索引（MEMORY.md）
- 主题文件完整保留
- 仍然可以访问完整记忆
- 只是部分索引不可见
