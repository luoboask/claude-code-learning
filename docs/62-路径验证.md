## 1. 路径验证概述

### 1.1 为什么需要路径验证？

**问题**: 用户可能通过路径操作访问或修改敏感文件。

**攻击示例**:
```bash
# 目录遍历攻击
../../etc/passwd

# 符号链接攻击
ln -s /etc/passwd /tmp/link
cat /tmp/link

# Unicode 绕过
\u002e\u002e\u002f etc/passwd  # ../etc/passwd
```

**解决**: 路径验证确保文件操作在允许的范围内。

## 2. 路径规范化

### 2.1 实现原理

```typescript
// source/src/utils/path.ts

export function normalizePath(path: string): string {
  // 1. 统一分隔符
  path = path.replace(/\\/g, '/')
  
  // 2. 解析 ..
  const segments = path.split('/')
  const result: string[] = []
  
  for (const segment of segments) {
    if (segment === '..') {
      result.pop()
    } else if (segment !== '.' && segment !== '') {
      result.push(segment)
    }
  }
  
  // 3. 重新组合
  return (path.startsWith('/') ? '/' : '') + result.join('/')
}
```

---

### 2.2 符号链接解析

```typescript
// source/src/utils/fsOperations.ts

export async function resolveSymlinks(path: string): Promise<string> {
  try {
    const realPath = await fs.realpath(path)
    return realPath
  } catch (error) {
    if ((error as NodeJS.ErrnoException).code === 'ENOENT') {
      // 文件不存在，返回规范化路径
      return normalizePath(path)
    }
    throw error
  }
}
```

### 2.3 路径验证函数

```typescript
// source/src/utils/permissions/pathValidation.ts

export async function validatePath(
  requestedPath: string,
  allowedPaths: string[],
  options: ValidatePathOptions = {},
): Promise<ValidationResult> {
  // 1. 规范化路径
  const normalizedPath = normalizePath(requestedPath)
  
  // 2. 解析符号链接
  const resolvedPath = await resolveSymlinks(normalizedPath)
  
  // 3. 检查允许列表
  const isAllowed = allowedPaths.some(allowedPath => {
    return isPathWithin(resolvedPath, allowedPath)
  })
  
  if (!isAllowed) {
    return {
      valid: false,
      message: `Path ${resolvedPath} is not within allowed paths`,
    }
  }
  
  // 4. 检查敏感路径
  if (isSensitivePath(resolvedPath)) {
    return {
      valid: false,
      message: `Path ${resolvedPath} is a sensitive path`,
    }
  }
  
  return { valid: true }
}

export function isPathWithin(path: string, parent: string): boolean {
  const normalizedPath = path.endsWith('/') ? path : path + '/'
  const normalizedParent = parent.endsWith('/') ? parent : parent + '/'
  
  return normalizedPath.startsWith(normalizedParent)
}
```

## 3. 常见路径攻击

### 3.1 目录遍历

**攻击示例**:
```bash
# 基本遍历
../../../etc/passwd

# URL 编码
%2e%2e%2f%2e%2e%2f%2e%2e%2fetc%2fpasswd

# Unicode 编码
\u002e\u002e\u002f etc/passwd

# 混合编码
..%2f..%2f..%2fetc/passwd
```

**防护**:
```typescript
function detectPathTraversal(path: string): boolean {
  // 检测 ..
  if (path.includes('..')) return true
  
  // 检测 URL 编码
  if (path.includes('%2e') || path.includes('%2E')) return true
  
  // 检测 Unicode 编码
  if (path.includes('\\u002e')) return true
  
  return false
}
```

---

### 3.2 符号链接攻击

**攻击示例**:
```bash
# 创建符号链接
ln -s /etc/passwd /tmp/allowed/link

# 通过允许的路径访问敏感文件
cat /tmp/allowed/link
```

**防护**:
```typescript
async function validateWithSymlinkCheck(path: string, allowedPaths: string[]): Promise<boolean> {
  // 解析符号链接
  const realPath = await fs.realpath(path)
  
  // 检查真实路径是否在允许范围内
  return allowedPaths.some(allowed => isPathWithin(realPath, allowed))
}
```

---

### 3.3 大小写绕过

**攻击示例**:
```bash
# Linux 大小写敏感
/etc/Passwd  # 不同于 /etc/passwd

# Windows 大小写不敏感
C:\Windows\System32  # 等同于 c:\windows\system32
```

**防护**:
```typescript
function normalizeCase(path: string, platform: string): string {
  if (platform === 'win32' || platform === 'darwin') {
    // Windows 和 macOS 大小写不敏感
    return path.toLowerCase()
  }
  return path
}
```

---

### 3.4 空字节注入

**攻击示例**:
```bash
# 空字节截断
/tmp/allowed/file.txt%00.jpg

# 在某些语言中，%00 会截断字符串
# 实际访问的是 /tmp/allowed/file.txt
```

**防护**:
```typescript
function detectNullByte(path: string): boolean {
  return path.includes('\0') || path.includes('%00')
}
```

## 4. 只读约束验证

### 4.1 只读路径检查

```typescript
// source/src/tools/BashTool/readOnlyValidation.ts

export async function checkReadOnlyConstraints(
  command: string,
  context: ToolUseContext,
): Promise<PermissionResult> {
  // 提取命令中的路径
  const paths = extractPathsFromCommand(command)
  
  // 检查每个路径
  for (const path of paths) {
    // 检查是否是写操作
    if (isWriteOperation(command, path)) {
      // 检查路径是否只读
      if (await isReadOnlyPath(path, context)) {
        return {
          behavior: 'ask',
          message: `Command attempts to write to read-only path: ${path}`,
        }
      }
    }
  }
  
  return { behavior: 'passthrough' }
}
```

---

### 4.2 只读路径判断

```typescript
async function isReadOnlyPath(path: string, context: ToolUseContext): Promise<boolean> {
  // 1. 检查系统目录
  const systemDirs = ['/etc', '/usr', '/bin', '/sbin', '/var']
  if (systemDirs.some(dir => path.startsWith(dir))) {
    return true
  }
  
  // 2. 检查用户配置
  const readOnlyPaths = context.options.readOnlyPaths || []
  if (readOnlyPaths.some(allowed => isPathWithin(path, allowed))) {
    return true
  }
  
  // 3. 检查文件权限
  try {
    const stats = await fs.stat(path)
    // 检查是否可写
    return !(stats.mode & 0o200)
  } catch {
    // 文件不存在，假设可写
    return false
  }
}
```

## 5. 路径验证实现

### 5.1 完整验证流程

```typescript
// source/src/utils/permissions/pathValidation.ts

export async function validatePathForTool(
  tool: Tool,
  input: Record<string, unknown>,
  context: ToolUseContext,
): Promise<PermissionResult> {
  // 1. 获取路径
  const path = getPathFromInput(tool, input)
  if (!path) {
    return { behavior: 'passthrough' }
  }
  
  // 2. 检测攻击模式
  if (detectPathTraversal(path)) {
    return {
      behavior: 'ask',
      message: 'Path contains directory traversal patterns',
    }
  }
  
  if (detectNullByte(path)) {
    return {
      behavior: 'ask',
      message: 'Path contains null byte',
    }
  }
  
  // 3. 规范化路径
  const normalizedPath = normalizePath(path)
  const resolvedPath = await resolveSymlinks(normalizedPath)
  
  // 4. 检查允许列表
  const allowedPaths = getAllowedPathsForTool(tool, context)
  const isAllowed = allowedPaths.some(allowed => isPathWithin(resolvedPath, allowed))
  
  if (!isAllowed) {
    return {
      behavior: 'ask',
      message: `Path ${resolvedPath} is not within allowed paths`,
    }
  }
  
  // 5. 检查敏感路径
  if (isSensitivePath(resolvedPath)) {
    return {
      behavior: 'ask',
      message: `Path ${resolvedPath} is a sensitive path`,
    }
  }
  
  // 6. 检查只读约束
  if (tool.name === 'FileWrite' || tool.name === 'FileEdit') {
    if (await isReadOnlyPath(resolvedPath, context)) {
      return {
        behavior: 'ask',
        message: `Cannot write to read-only path: ${resolvedPath}`,
      }
    }
  }
  
  return { behavior: 'allow' }
}
```

---

### 5.2 敏感路径列表

```typescript
const SENSITIVE_PATHS = [
  '/etc/passwd',
  '/etc/shadow',
  '/etc/sudoers',
  '/root',
  '/boot',
  '/proc',
  '/sys',
  '/dev',
]

function isSensitivePath(path: string): boolean {
  return SENSITIVE_PATHS.some(sensitive => 
    path === sensitive || path.startsWith(sensitive + '/')
  )
}
```

## 6. 最佳实践

### 6.1 路径验证

```markdown
✅ 应该做:

1. 始终规范化路径
   - 解析 ..
   - 解析符号链接
   - 统一分隔符

2. 使用允许列表
   - 明确定义允许的目录
   - 使用 isPathWithin 检查

3. 检查敏感路径
   - 系统目录
   - 敏感配置文件

4. 记录路径操作
   - 审计所有文件访问
   - 监控异常模式

❌ 不应该做:

1. 信任用户输入的路径
2. 跳过符号链接解析
3. 忽视大小写问题
4. 允许访问系统目录
```

}
