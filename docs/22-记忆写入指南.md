## 1. 写入流程

### 1.1 两步流程

```
保存记忆 = 两步过程

Step 1 — 将记忆写入独立文件
  例如：user_role.md, feedback_testing.md
  使用 frontmatter 格式

Step 2 — 在 MEMORY.md 中添加指针
  MEMORY.md 是索引，不是记忆本身
  每条目一行，<150 字符
```

### 1.2 流程图

```
用户请求保存记忆
    │
    ↓
┌─────────────────────────────────┐
│ Step 1: 创建记忆文件             │
│ - 选择类型 (user/feedback/...)  │
│ - 创建文件 (feedback_xxx.md)    │
│ - 写入 frontmatter              │
│ - 写入内容                      │
└───────────────┬─────────────────┘
                │
                ↓
┌─────────────────────────────────┐
│ Step 2: 更新 MEMORY.md 索引      │
│ - 添加指针条目                   │
│ - 格式：- [Title](file.md) — hook│
│ - 保持 <150 字符                 │
└───────────────┬─────────────────┘
                │
                ↓
┌─────────────────────────────────┐
│ 完成                             │
│ - 记忆已保存                     │
│ - 下次会话自动加载               │
└─────────────────────────────────┘
```

## 2. Frontmatter 格式

### 2.1 标准格式

```markdown
---
name: {{记忆名称}}
description: {{一行描述 — 用于决定未来会话中的相关性，要具体}}
type: {{user|feedback|project|reference}}
---

{{记忆内容}}
```

### 2.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | string | ✅ | 简短描述性名称 |
| `description` | string | ✅ | 一行描述，用于相关性判断 |
| `type` | enum | ✅ | 4 种类型之一 |
| `created` | date | ❌ | 创建日期（可选） |
| `updated` | date | ❌ | 最后更新日期（可选） |
| `scope` | enum | ❌ | private/team（feedback/project 需要） |

### 2.3 示例

```markdown
# User Memory 示例
---
name: 数据科学家角色
description: 用户是数据科学家，专注于可观测性和日志分析
type: user
created: 2026-04-02
---

用户是数据科学家，目前专注于可观测性/logging 领域。
```

```markdown
# Feedback Memory 示例
---
name: 测试必须用真实数据库
description: 集成测试禁止 mock 数据库，必须使用真实连接
type: feedback
scope: team
created: 2026-04-02
---

集成测试必须 hit 真实数据库，不能使用 mock。

**Why:** 去年 mock 测试通过但生产迁移失败

**How to apply:** 所有集成测试必须配置真实数据库连接
```

```markdown
# Project Memory 示例
---
name: 移动端发布冻结合并
description: 2026-03-05 后冻结合并，移动端团队切割发布分支
type: project
scope: team
---

2026-03-05 后冻结合并（移动端发布分支切割）

**Why:** 移动端团队要切割 v2.0 发布分支

**How to apply:** 标记所有计划在此日期后的非关键 PR 工作
```

```markdown
# Reference Memory 示例
---
name: Linear 管道 bug 追踪
description: 管道 bug 在 Linear 项目"INGEST"中追踪
type: reference
scope: team
---

管道 bug 在 Linear 项目"INGEST"中追踪

**URL:** https://linear.app/company/project/INGEST
```

## 3. 内容结构

### 3.1 Feedback/Project 结构

```markdown
---
name: {{名称}}
description: {{描述}}
type: {{feedback|project}}
---

{{规则/事实本身}}

**Why:** {{原因 — 过去的事件或强烈偏好}}

**How to apply:** {{何时/何地应用此指导}}
```

### 3.2 为什么需要 Why 和 How

| 部分 | 作用 | 示例 |
|------|------|------|
| **规则本身** | 说明要做什么/不做什么 | "集成测试必须用真实数据库" |
| **Why** | 解释原因，便于判断边界情况 | "去年 mock 测试通过但生产迁移失败" |
| **How to apply** | 说明何时应用 | "所有集成测试必须配置真实数据库连接" |

### 3.3 完整示例

```markdown
---
name: 不要 mock 数据库
description: 集成测试禁止 mock 数据库，必须使用真实连接
type: feedback
scope: team
---

集成测试必须 hit 真实数据库，不能使用 mock。

**Why:** 
去年 Q4 发布时，mock 的数据库测试全部通过，但生产迁移因字段类型不匹配失败，
导致 2 小时停机。根本原因：mock 层没有捕获 schema 变更。

**How to apply:**
- 所有集成测试必须配置真实数据库连接
- 使用测试数据库（非生产）
- CI/CD 配置中启用真实数据库
- 单元测试可以 mock
- 端到端测试必须真实环境

**例外:**
- 外部 API 依赖可以 mock（如果 API 不稳定）
- 性能测试可以 mock（如果需要控制变量）
```

## 4. 写入规则

### 4.1 应该做

```markdown
✅ 应该做:

1. 为每个记忆创建独立文件
   - user_role.md
   - feedback_testing.md
   - project_deadline.md
   - reference_linear.md

2. 使用 frontmatter 格式
   - name, description, type 必填
   - 根据需要添加 created, updated, scope

3. 保持名称、描述、类型字段与内容同步
   - 更新内容时同时更新 frontmatter

4. 按主题组织记忆，而非按时间顺序
   - feedback_testing.md（主题）
   - 而非 feedback_2026-04-02.md（时间）

5. 更新或删除错误/过时的记忆
   - 发现记忆错误时立即更新
   - 记忆不再适用时删除或标记为过时

6. 写入前检查是否有重复记忆可更新
   - 先搜索现有记忆
   - 有相关记忆则更新而非新建
```

### 4.2 不应该做

```markdown
❌ 不应该做:

1. 直接在 MEMORY.md 中写记忆内容
   - MEMORY.md 只是索引
   - 内容应该在独立文件中

2. 索引条目超过 150 字符
   - 保持简洁：- [Title](file.md) — 一行钩子

3. 写入可推导的内容
   - 代码模式（grep 可推导）
   - Git 历史（git log 是权威）
   - 架构（阅读代码可推导）

4. 写入临时任务详情
   - 进行中的工作
   - 临时状态
   - 当前对话上下文
```

## 5. MEMORY.md 索引格式

### 5.1 标准格式

```markdown
- [记忆名称](文件.md) — 一行钩子
```

### 5.2 示例

```markdown
# MEMORY.md

## User Memories
- [数据科学家角色](user_role.md) — 专注于可观测性/logging
- [Go 专家前端新手](user_expertise.md) — 用后端类比解释前端

## Feedback Memories
- [测试必须用真实数据库](feedback_testing.md) — 集成测试禁止 mock
- [简洁回复无总结](feedback_communication.md) — 不要结尾总结

## Project Memories
- [移动端发布冻结合并](project_freeze.md) — 2026-03-05 后冻结
- [认证中间件重写](project_auth.md) — 法律合规要求驱动

## Reference Memories
- [Linear 管道 bug 追踪](reference_linear.md) — 项目"INGEST"
- [Grafana 延迟仪表板](reference_grafana.md) — oncall 监控
```

### 5.3 字符限制

```markdown
# ✅ 好的索引条目（<150 字符）
- [测试必须用真实数据库](feedback_testing.md) — 集成测试禁止 mock，去年生产事故

# ❌ 坏的索引条目（>150 字符）
- [测试必须用真实数据库](feedback_testing.md) — 集成测试禁止 mock 数据库，必须使用真实连接，因为去年 mock 测试通过但生产迁移失败，导致 2 小时停机，这是一个项目测试政策
  # 太长！应该简化为：集成测试禁止 mock，去年生产事故
```

## 6. 文件命名规范

### 6.1 命名格式

```
<type>_<topic>.md

type: user|feedback|project|reference
topic: 简短主题描述（小写，下划线分隔）
```

### 6.2 示例

```markdown
# User Memory
user_role.md
user_expertise.md
user_preferences.md

# Feedback Memory
feedback_testing.md
feedback_communication.md
feedback_workflow.md

# Project Memory
project_freeze.md
project_auth.md
project_deadline.md

# Reference Memory
reference_linear.md
reference_grafana.md
reference_slack.md
```

### 6.3 避免的命名

```markdown
# ❌ 避免时间命名
feedback_2026-04-02.md  # 应该按主题命名

# ❌ 避免过长命名
feedback_integration_tests_must_use_real_database_not_mock.md  # 太长了

# ❌ 避免模糊命名
memory1.md  # 不清晰
temp.md  # 临时文件应该删除
```

## 7. 写入时机

### 7.1 立即保存

```markdown
✅ 立即保存:

1. 用户明确要求记住
   - "记住这个"
   - "别忘了"
   - "保存下来"

2. 重要的用户细节
   - 角色、偏好、知识
   - 技术栈、经验水平

3. 工作方式指导
   - 纠正 ("不要...")
   - 确认 ("对，正是这样")

4. 项目关键信息
   - 截止日期
   - 负责人
   - 技术决策
```

### 7.2 延迟保存

```markdown
⏸️ 可以延迟:

1. 临时任务状态
   - 等待用户确认后再保存

2. 不确定的信息
   - "可能是..." → 确认后再保存

3. 敏感信息
   - 先询问用户是否应该保存
```

### 7.3 不保存

```markdown
❌ 不保存:

1. 可推导的信息
   - 代码模式、架构、Git 历史

2. 临时状态
   - 进行中的工作、当前对话上下文

3. CLAUDE.md 已有的内容
   - 避免重复
```

## 8. 更新和删除

### 8.1 何时更新

```markdown
✅ 应该更新:

1. 记忆内容错误
   - 发现事实错误时立即更新

2. 记忆过时
   - 项目状态变化
   - 用户角色变化

3. 记忆不完整
   - 添加缺失的 Why 或 How
```

### 8.2 何时删除

```markdown
✅ 应该删除:

1. 记忆不再适用
   - 项目已结束
   - 用户偏好变化

2. 重复记忆
   - 合并重复的记忆

3. 错误记忆
   - 保存了不应该保存的内容
```

### 8.3 如何更新

```markdown
# 更新步骤

1. 打开记忆文件
2. 更新内容
3. 更新 frontmatter 的 updated 字段
4. 更新 MEMORY.md 索引（如需要）

# 示例

---
name: 测试必须用真实数据库
description: 集成测试禁止 mock 数据库，必须使用真实连接
type: feedback
scope: team
created: 2026-04-02
updated: 2026-04-02  # 添加更新日期
---

# 更新内容...
```

## 9. 常见错误

### 9.1 直接在 MEMORY.md 中写内容

```markdown
# ❌ 错误
## MEMORY.md

- 用户是数据科学家  # 不应该直接写内容
- 测试必须用真实数据库

# ✅ 正确
## MEMORY.md

- [数据科学家角色](user_role.md) — 专注于可观测性
- [测试必须用真实数据库](feedback_testing.md) — 集成测试禁止 mock
```

### 9.2 索引条目太长

```markdown
# ❌ 错误
- [测试必须用真实数据库](feedback_testing.md) — 集成测试禁止 mock 数据库，必须使用真实连接，因为去年 mock 测试通过但生产迁移失败，导致 2 小时停机，这是一个项目测试政策，每个贡献者都应该遵循

# ✅ 正确
- [测试必须用真实数据库](feedback_testing.md) — 集成测试禁止 mock，去年生产事故
```

### 9.3 保存可推导信息

```markdown
# ❌ 错误
---
name: 项目架构
type: project
---

项目使用分层架构：
- controllers/ 处理请求
- services/ 业务逻辑
- models/ 数据访问

# ✅ 正确做法：不要保存，阅读代码即可
```
