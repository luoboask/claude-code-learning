## 🎯 学习目标

完成本节后，你将能够：
- ✅ 区分 4 种记忆类型
- ✅ 正确选择记忆类型

## 1. User Memory（用户记忆）

### 1.1 定义

**User Memory** 包含关于用户角色、目标、职责和知识的信息。

**作用域**: 始终私有 (always private)

### 1.2 何时保存

```
✅ 应该保存:
- 用户的角色和职位
- 用户的专业领域
- 用户的偏好（沟通风格、工具偏好等）
- 用户的学习目标
- 用户的技术栈

❌ 不应该保存:
- 临时的工作任务
- 项目特定的信息（属于 Project Memory）
- 用户的反馈指导（属于 Feedback Memory）
```

### 1.3 保存时机

| 用户表述 | 记忆类型 | 示例 |
|----------|----------|------|
| "我是..." | User | "我是数据科学家" |
| "我负责..." | User | "我负责后端开发" |
| "我偏好..." | User | "我偏好 TypeScript" |
| "我想学习..." | User | "我想学习 Rust" |

### 1.4 完整示例

```markdown
---
name: 数据科学家角色
description: 用户是数据科学家，专注于可观测性和日志分析
type: user
---

用户是数据科学家，目前专注于可观测性/logging 领域。

**技术背景:**
- 精通 Python 和 SQL
- 熟悉数据管道和 ETL
- 有机器学习项目经验

**沟通偏好:**
- 喜欢详细的技术解释
- 偏好代码示例
- 重视性能和安全考虑

**如何应用:**
解释问题时应该：
- 使用数据分析相关的类比
- 强调日志和监控的最佳实践
- 提供可量化的结果
- 优先推荐 Python 解决方案
```

### 1.5 保存示例

```
用户：我在调查我们有哪些日志记录
助手：[保存私有用户记忆：用户是数据科学家，目前专注于可观测性/logging]

用户：我写了十年 Go 但第一次接触这个 repo 的前端
助手：[保存私有用户记忆：Go 专家，前端新手 — 用后端类比解释前端概念]

用户：我是团队的技术负责人，负责架构决策
助手：[保存私有用户记忆：技术负责人角色，负责架构决策 — 提供战略层面的建议]
```

## 2. Feedback Memory（反馈记忆）

### 2.1 定义

**Feedback Memory** 记录用户给出的工作方式指导——既要避免什么，也要保持什么。

**作用域**: 默认私有。仅当指导是项目级约定时才保存为团队

### 2.2 何时保存

```
✅ 应该保存:
- 用户纠正你的方法 ("不要...", "停止...")
- 用户确认非显而易见的方法有效 ("对，正是这样")
- 测试和开发约定
- 沟通偏好
- 工作流程指导

❌ 不应该保存:
- 个人风格偏好（除非明确为团队约定）
- 临时的工作安排
- 代码模式（可推导）
```

### 2.3 保存时机

| 用户表述 | 记忆类型 | 示例 |
|----------|----------|------|
| "不要..." | Feedback | "不要 mock 数据库" |
| "停止..." | Feedback | "停止总结 diff" |
| "对，正是这样" | Feedback | "对，这种重构方式正确" |
| "我们总是..." | Feedback | "我们总是先写测试" |

### 2.4 完整示例

```markdown
---
name: 测试必须用真实数据库
description: 集成测试禁止 mock 数据库，必须使用真实连接
type: feedback
---

集成测试必须 hit 真实数据库，不能使用 mock。

**Why:** 
去年 mock 测试通过但生产迁移失败，mock/生产差异掩盖了损坏的迁移。
具体事件：Q4 发布时，mock 的数据库测试全部通过，但生产迁移因字段类型不匹配失败，
导致 2 小时停机。

**How to apply:**
- 所有集成测试必须配置真实数据库连接
- 使用测试数据库（非生产）
- 单元测试可以 mock
- 端到端测试必须真实环境
- CI/CD 配置中启用真实数据库

**作用域:** 团队
这是项目测试政策，不是个人偏好。每个贡献者都应该遵循。
```

### 2.5 保存示例

```
用户：不要在这些测试中 mock 数据库——去年我们因此吃亏
助手：[保存团队反馈记忆：集成测试必须用真实数据库。原因：mock/生产差异掩盖了损坏的迁移]

用户：停止在每次回复后总结你做了什么，我能看懂 diff
助手：[保存私有反馈记忆：用户想要简洁回复，不要结尾总结]

用户：对，这种重构方式正确，拆分这个 PR 会是多余的
助手：[保存私有反馈记忆：此区域的重构偏好单个 bundled PR，而非多个小 PR]
```

## 3. Project Memory（项目记忆）

### 3.1 定义

**Project Memory** 记录关于项目中正在进行的工作、目标、计划、bug 或事件的信息。

**作用域**: 私有或团队，但强烈偏向团队

### 3.2 何时保存

```
✅ 应该保存:
- 正在进行的工作和负责人
- 项目目标和截止日期
- 技术决策和原因
- Bug 和事件
- 利益相关者要求

❌ 不应该保存:
- 临时的任务状态（属于 Tasks）
- 代码实现细节（可推导）
- Git 历史（git log 是权威）
```

### 3.3 保存时机

| 用户表述 | 记忆类型 | 示例 |
|----------|----------|------|
| "我们在做..." | Project | "我们在重构认证中间件" |
| "截止日期是..." | Project | "截止日期是 2026-03-05" |
| "原因是..." | Project | "原因是法律合规要求" |
| "X 负责..." | Project | "张三负责移动端发布" |

### 3.4 完整示例

```markdown
---
name: 移动端发布冻结合并
description: 2026-03-05 后冻结合并，移动端团队切割发布分支
type: project
---

2026-03-05 后冻结合并（移动端发布分支切割）

**Why:**
移动端团队要切割 v2.0 发布分支，需要代码稳定。
任何非关键合并都可能导致回归，延迟发布。

**How to apply:**
- 标记所有计划在此日期后的非关键 PR 工作
- 关键 bug 修复仍需立即合并（需要技术负责人批准）
- 提前通知用户可能的延迟
- 2026-03-10 后解冻（预计）

**负责人:** 李四（移动端团队）
**截止日期:** 2026-03-05
**状态:** 进行中
```

### 3.5 保存示例

```
用户：周四后我们要冻结所有非关键合并——移动端团队要切割发布分支
助手：[保存项目记忆：2026-03-05 后冻结合并。标记该日期后的非关键 PR 工作]

用户：我们重写认证中间件的原因是法律团队标记了会话 token 存储不合规
助手：[保存项目记忆：认证中间件重写由法律/合规要求驱动，而非技术债务 — 范围决策应 favor 合规而非体验]

用户：张三负责这个功能，他下周二前完成
助手：[保存项目记忆：张三负责该功能，截止日期 2026-03-12]
```

## 4. Reference Memory（参考记忆）

### 4.1 定义

**Reference Memory** 存储指向外部系统中信息位置的指针。

**作用域**: 通常是团队

### 4.2 何时保存

```
✅ 应该保存:
- Bug 追踪系统位置
- 监控仪表板
- 文档位置
- Slack 频道
- 外部 API 文档

❌ 不应该保存:
- 内部代码位置（可搜索）
- 临时链接
```

### 4.3 保存时机

| 用户表述 | 记忆类型 | 示例 |
|----------|----------|------|
| "在...找" | Reference | "在 Linear 项目'INGEST'找" |
| "...是追踪" | Reference | "Grafana 是追踪延迟的" |
| "文档在..." | Reference | "文档在 Notion" |
| "去...问" | Reference | "去#backend 频道问" |

### 4.4 完整示例

```markdown
---
name: Linear 管道 bug 追踪
description: 管道 bug 在 Linear 项目"INGEST"中追踪
type: reference
---

管道 bug 在 Linear 项目"INGEST"中追踪

**URL:** https://linear.app/company/project/INGEST

**用途:**
- 所有数据管道 bug 都记录在此
- 包含 bug 描述、优先级、分配
- 与数据团队共享

**如何应用:**
当用户提到管道 bug 或数据问题时：
1. 先检查 Linear "INGEST" 项目
2. 查看是否有已知问题
3. 更新现有 ticket 或创建新的
```

### 4.5 保存示例

```
用户：如果你想了解这些 ticket 的背景，查看 Linear 项目"INGEST"，我们在那里追踪所有管道 bug
助手：[保存参考记忆：管道 bug 在 Linear 项目"INGEST"中追踪]

用户：Grafana 仪表板 grafana.internal/d/api-latency 是 oncall 监控的
助手：[保存参考记忆：grafana.internal/d/api-latency 是 oncall 延迟仪表板 — 编辑请求路径代码时检查]

用户：团队文档在 Notion 的"Engineering/Backend"页面
助手：[保存参考记忆：团队文档在 Notion "Engineering/Backend" 页面]
```

## 5. 类型选择决策树

```
用户说了什么？
    │
    ├─→ "我是..." / "我偏好..."
    │       └─→ User Memory
    │
    ├─→ "不要..." / "停止..." / "对，正是这样"
    │       └─→ Feedback Memory
    │
    ├─→ "我们在做..." / "截止日期..." / "原因是..."
    │       └─→ Project Memory
    │
    └─→ "在...找" / "...是追踪" / "文档在..."
            └─→ Reference Memory
```

## 6. 常见错误

### 6.1 类型混淆

```markdown
❌ 错误：将 Feedback 保存为 User

# 错误示例
---
name: 测试偏好
type: user  # ❌ 应该是 feedback
---

用户偏好集成测试用真实数据库。

# 正确示例
---
name: 测试必须用真实数据库
type: feedback  # ✅ 这是工作方式指导
---

集成测试必须 hit 真实数据库。
```

### 6.2 作用域错误

```markdown
❌ 错误：将个人偏好保存为团队

# 错误示例
---
name: 回复风格
type: feedback
---

用户想要简洁回复，不要结尾总结。
**作用域:** 团队  # ❌ 这是个人沟通偏好，应该是私有

# 正确示例
**作用域:** 私有  # ✅ 这是个人偏好
```

### 6.3 保存可推导信息

```markdown
❌ 错误：保存代码模式

# 错误示例
---
name: 项目架构
type: project  # ❌ 这可以通过阅读代码获得
---

项目使用分层架构：
- controllers/ 处理请求
- services/ 业务逻辑
- models/ 数据访问

# 正确做法：不要保存，阅读代码即可
```

## 7. Frontmatter 规范

### 7.1 必填字段

```yaml
---
name: 记忆名称  # 简短描述性名称
description: 一行描述 — 用于决定未来会话中的相关性，要具体
type: user|feedback|project|reference  # 4 种类型之一
---
```

### 7.2 可选字段

```yaml
---
name: 记忆名称
description: 一行描述
type: feedback
created: 2026-04-02  # 创建日期（可选）
updated: 2026-04-02  # 最后更新日期（可选）
scope: private|team  # 作用域（可选，feedback/project 需要）
---
```

### 7.3 内容结构

```markdown
---
name: 记忆名称
description: 一行描述
type: feedback
---

**规则/事实本身**

**Why:** 原因（过去的事件或强烈偏好）

**How to apply:** 何时/何地应用此指导
```
